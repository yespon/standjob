## 二、关键差异 ❌

### D1: 【重要】双线推进逻辑未实现

**Skill 要求（SKILL.md:75-92）**：
```
两条独立的引导线：
- 线1：客户 → 价值 → 效能
- 线2：任务 → 目的 → 成果

**严禁把任务和效能挂钩**，它们是两条完全独立的线
推进顺序：严格先线1，再线2，不穿插、不跳跃
线1全部过关后，再进入线2
```

**当前实现**（nodes.py:810-817）：
```python
CATEGORY_ORDER = {
    "客户定义": 0,
    "岗位价值": 1,
    "岗位效能": 2,
    "岗位任务": 3,
    "任务目的与成果": 4,
}
```

**问题**：
- 虽然排序正确，但没有**强制线1完成后才进入线2**的机制
- 如果"客户定义"、"岗位价值"、"岗位效能"都有问题，当前会逐个引导
- 但如果线1没完成就跳到线2（比如岗位任务），不符合 skill 要求

**修复建议**：
添加 `current_line` 状态字段，明确区分线1/线2，线1全部 resolved 后才允许进入线2。

---

### D2: 【重要】任务与效能分离未明确

**Skill 要求（SKILL.md:77-78）**：
```
严禁把任务和效能挂钩，它们是两条完全独立的线
```

**当前实现**：
- `issue_queue` 包含所有问题项，按 category 排序
- 没有显式区分线1（客户/价值/效能）和线2（任务/目的/成果）

**影响**：引导时可能无意中把任务和效能关联讨论，违反 skill 铁律。

---

### D3: 辅助任务特殊处理未实现

**Skill 要求（SKILL.md:96-102）**：
```
辅助任务的目的和成果可选填，不强求完整。
- 核心任务：必须有明确的目的和成果
- 辅助任务：目的和成果可以简化或省略，不要追问
```

**当前实现**：
- `_pick_related_fields()` 和 `_should_relax_issue()` 没有区分核心/辅助任务
- 所有任务都按同一标准检查

**修复建议**：
添加任务类型识别（核心/辅助），对辅助任务放宽目的/成果检查。

---

### D4: 反模式 #10 检查不足

**Skill 要求**：不展示打分/评审/过关信息

**当前实现问题**：
- `review_item` 节点有 `print(f"...得分{score}")` 调试输出
- `score` 字段在 state 中保留（虽然用户不可见）

## 三、建议修复优先级

| 优先级 | 问题 | 影响 | 修复复杂度 |
|-------|------|------|----------|
| **P0** | D1 双线推进逻辑 | 核心流程不符合 skill | 中（需加状态字段） |

| **P1** | D3 辅助任务处理 | 过度诊断辅助任务 | 中（需区分任务类型） |
| **P1** | D4 清理评分信息 | 反模式 #10/#11 | 低（删除 debug 日志） |

---

## 四、修复方案

### 方案：添加双线推进状态

在 `CoachState` 中添加：
```python
current_line: int  # 1 = 线1(客户-价值-效能), 2 = 线2(任务-目的-成果)
line1_completed: bool  # 线1是否全部完成
```

在 `process_response` 中：
- 检查当前问题项属于线1还是线2
- 如果线1未完成，禁止进入线2
- 线1全部 resolved 后，明确告知用户进入下一阶段

### 方案：区分核心/辅助任务

在 `state.py` 中添加任务类型字段，在评审时根据任务类型调整严格度。
